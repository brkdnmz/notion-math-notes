Remainders to Help Us Miserable C++’ers!

If we can’t deal with huge integers, let remainders help us: Modular Arithmetic.

We know what a remainder is: it is the value produced after two integers are divided. For instance, we get 222﻿ as the remainder when 171717﻿ is divided by 555﻿, or 000﻿ when 369369369﻿ is divided by 123123123﻿.

There was the modulo operator %, particularly for finding the remainder, ignoring the division result. For the examples above, 17 % 5=217 \ \% \ 5 = 217 % 5=2﻿ and 369 % 123=0369 \ \% \ 123 = 0369 % 123=0﻿.

To deal with large integers, we use this operator to compute the answer modulo some integer mmm﻿, where mmm﻿ is called the modulus. We don’t refer to mmm﻿ alone, instead, we often use the expression “modulo mmm﻿”. This is why the term modulus isn’t used so much.

What does “xxx﻿ modulo mmm﻿” even mean?

It is how we refer to the remainder itself since the remainder is the result of this modulo operation. It means the same as “remainder of xxx﻿ divided by mmm﻿”, “remainder when xxx﻿ is divided by mmm﻿”…

But wait, what if mmm﻿ is so small that even if our calculations are wrong, we get the same remainder as the true answer’s remainder? For instance, 131313﻿ and 181818﻿ produce the same remainder modulo 555﻿, but we should’ve computed 131313﻿ instead of 181818﻿.

Yes, that’s so right. This is why mmm﻿ is never that small. 😏

To decrease the probability of that happening, a large prime is chosen for mmm﻿ such as 109+710^9 + 7109+7﻿, 998244353998244353998244353﻿, etc.

Of course, there are so many other primes, but these two numbers have some special properties on their own, which even I don’t completely know about.

FYI, we almost always use 109+710^9 + 7109+7﻿.

Well, so far so good. But our calculations will consist of many steps that lead to the final result, therefore many intermediate values. We can’t just find the final answer in one step, and then find the remainder. But, we should somehow always deal with small integers, as this is our actual purpose.

Then, how do we safely focus on finding the remainder?

For instance, calculating 5!5!5!﻿ requires several steps:

  1. Start with 111﻿.
  1. Multiply by 222﻿ to get 2!=22! = 22!=2﻿.
  1. Multiply by 333﻿ to get 3!=63! = 63!=6﻿.
  1. Multiply by 444﻿ to get 4!=244! = 244!=24﻿.
  1. Finally, multiply by 555﻿ to get 5!=1205! = 1205!=120﻿.

Suppose we want to calculate 5!5!5!﻿ modulo 232323﻿.

What should we actually do in previous steps to immediately obtain the remainder right after the final step, which is 120 % 23=5120 \ \% \ 23 = 5120 % 23=5﻿?

We might consider calculating the remainder at each step, and continue the calculation with it:

  1. 1⋅2=2,2 % 23=21 \cdot 2 = 2, 2 \ \% \ 23 = 21⋅2=2,2 % 23=2﻿
  1. 2⋅3=6,6 % 23=62 \cdot 3 = 6, 6 \ \% \ 23 = 62⋅3=6,6 % 23=6﻿
  1. 6⋅4=24,24 % 23=16 \cdot 4 = 24, 24 \ \% \ 23 = 16⋅4=24,24 % 23=1﻿
  1. 1⋅5=5,5 % 23=51 \cdot 5 = 5, 5 \ \% \ 23 = 51⋅5=5,5 % 23=5﻿

Hey, we actually calculated the desired result!

But, is it always gonna work?

To answer this, let’s discover modular arithmetic.

Modular Arithmetic & Congruence

Let’s move on with our modulus mmm﻿. Think about the integers which produce the same remainder divided by mmm﻿. For example, when m=3m = 3m=3﻿, each of the integers 2,5,8,11,…2, 5, 8, 11, \dots2,5,8,11,…﻿ yields a remainder of 222﻿ when divided by 333﻿. As it can be realized, the difference between adjacent numbers is always 333﻿. Why? Think about the division equation n=kd+rn = kd + rn=kd+r﻿. These numbers are in the form of 3k+23k + 23k+2﻿, and incrementing/decrementing kkk﻿ gives us the neighboring integers: if k=2k = 2k=2﻿, then we have 888﻿; for k=1k = 1k=1﻿ and 333﻿, we have the neighbors 555﻿ and 111111﻿.

To specify the common property of these integers 2,5,8,11,14,…2, 5, 8, 11, 14, \dots2,5,8,11,14,…﻿, we say that “they are congruent modulo 333﻿”, and we name this relation “congruence”.

You can think of congruence as the equality of remainders, just like how we say two numbers are equal if their values are the same — several numbers are congruent modulo mmm﻿ if their remainders are the same.

Similar to how we often compare two numbers when we talk about equality, we also compare two numbers when we talk about congruence. The relation “aaa﻿ is congruent to bbb﻿ modulo mmm﻿” is notated as follows:

That is, the same remainder is obtained when aaa﻿ or bbb﻿ is divided by mmm﻿. The ≡\equiv≡﻿ sign (equivalency) denotes congruence.

Negative numbers are also involved in the congruence domain. Note that a negative integer can also be carried into the range [0,m)[0, m)[0,m)﻿ by some number of additions, that is, its remainder is also defined.

Alright, I think we all got it.

On the other hand, remember that a mod ma \bmod mamodm﻿ refers to the remainder itself.
An integer’s remainder is the smallest positive integer it is congruent to.

Now, let’s recall when two integers aaa﻿ and bbb﻿ are congruent modulo mmm﻿: the remainder is the same when divided by mmm﻿.

So, we can express these numbers as a=pm+ra = pm + ra=pm+r﻿ and b=qm+rb = qm + rb=qm+r﻿.

Will this congruence be preserved when we subtract mmm﻿ from / add mmm﻿ to either aaa﻿ or bbb﻿ (or both)?

  • If we add mmm﻿, mmm﻿’s coefficient (ppp﻿ or qqq﻿) gets incremented.
  • If we subtract mmm﻿, mmm﻿’s coefficient gets decremented.

In both cases, the remainder rrr﻿ remains the same, so is congruence: they stay congruent.

This might seem pretty useless and unimportant, just like 2+2=42 + 2 = 42+2=4﻿.

But, we are going to derive some very handy properties just using this observation. This is how math works… Math is just so brilliant! 🤩

We have only dealt with single integers so far. Let’s play with expressions, starting with summations.

Summation modulo mmm﻿

Say we have nnn﻿ not-so-small integers a1,a2,a3,…,ana_1, a_2, a_3, \dots, a_na1​,a2​,a3​,…,an​﻿, and we want to calculate (a1+a2+a3+⋯+an) mod m.(a_1 + a_2 + a_3 + \dots + a_n )\bmod m.(a1​+a2​+a3​+⋯+an​)modm.﻿

Can we apply some operations while keeping the result (remainder) the same?

Of course! We can freely subtract mmm﻿ from whichever aia_iai​﻿ we want (or add mmm﻿ if it’s negative).

Let’s suppose we applied the necessary operations to every aia_iai​﻿ so that 0≤ai<m0 \le a_i < m0≤ai​<m﻿ always holds.

What we actually did is replace aia_iai​﻿’s with ai mod ma_i \bmod mai​modm﻿, i.e., their remainder when divided by mmm﻿.

So far so good: each aia_iai​﻿ became the smallest congruent positive integer. But the sum may get so big if we just keep adding.

Well, we can apply the same operation after each addition since this operation always preserves congruence.

Let’s continue with multiplication.

Multiplication modulo mmm﻿

Now, we want to calculate a1⋅a2⋅a3⋯an mod ma_1 \cdot a_2 \cdot a_3 \cdots a_n \bmod ma1​⋅a2​⋅a3​⋯an​modm﻿.

The same idea comes to mind: addition to/subtraction from aia_iai​﻿ (changing aia_iai​﻿ in short). But when we do that, the value of the expression does not change by just mmm﻿. So, is it even legal?

No, Master Chief, it’s not!

The result doesn’t change by mmm﻿, but it changes by some multiple of mmm﻿ because when we change aia_iai​﻿, the result changes by k⋅mk \cdot mk⋅m﻿ where kkk﻿ is the product of other aj (j≠i)a_j \ (j \ne i)aj​ (j=i)﻿.

The congruence is still preserved because changing the result by k⋅mk \cdot mk⋅m﻿ does not affect the remainder, you can think of it as adding/subtracting mmm﻿ several times.

So, whenever we change an aia_iai​﻿, the remainder stays the same.

As a result, everything valid for summation is also valid for multiplication!
Summations and multiplications combined

Any complex expression involving summation, multiplication, and nested expressions (parentheses) can be expanded to the general expression of a1⋆a2⋆a3⋆⋯⋆ana_1 \star a_2 \star a_3 \star \dots \star a_na1​⋆a2​⋆a3​⋆⋯⋆an​﻿, where every ⋆\star⋆﻿ is either +++﻿ or ⋅\cdot⋅﻿.

This expression can be seen as a summation of several multiplications. For instance, a1⋅a2+a3+a4⋅a5⋅a6a_1 \cdot a_2 + a_3 + a_4 \cdot a_5 \cdot a_6a1​⋅a2​+a3​+a4​⋅a5​⋅a6​﻿ is the summation of M1=a1⋅a2M_1 =a_1 \cdot a_2M1​=a1​⋅a2​﻿, M2=a3M_2 = a_3M2​=a3​﻿, and M3=a4⋅a5⋅a6M_3 = a_4 \cdot a_5 \cdot a_6M3​=a4​⋅a5​⋅a6​﻿.

Then, we can calculate Mi mod mM_i \bmod mMi​modm﻿ one by one, and what we are left afterward is just the summation of 333﻿ integers.

As a result, the same operation of changing aia_iai​﻿ holds also for these kinds of expressions!
Division

Things get a bit complicated when division is the case.

Suppose we want to calculate pq mod m\dfrac{p}{q} \bmod mqp​modm﻿. If we first attempt to find p mod mp \bmod mpmodm﻿, then divide it by qqq﻿, we won’t achieve the desired result. Cases where m−1<qm-1 < qm−1<q﻿ are a nice example.

Is there really no way, though? Let’s figure it out.

Let pq=F=km+r\dfrac{p}{q} = F = km + rqp​=F=km+r﻿ where rrr﻿ is the remainder (suppose FFF﻿ is an integer).

Then, p=qkm+qrp = qkm + qrp=qkm+qr﻿.

Notice that if m>rm > rm>r﻿, then p mod qm=qrp \bmod qm = qrpmodqm=qr﻿, and dividing this by qqq﻿ directly results in rrr﻿ itself. This is a quite practical method, which I first saw in someone else’s code.

Another method uses the equality p mod m=qr mod m=xp \bmod m = qr \bmod m = xpmodm=qrmodm=x﻿. Since x≡qr(modm)x \equiv qr \pmod mx≡qr(modm)﻿, we can definitely obtain qrqrqr﻿ itself by adding several mmm﻿’s to xxx﻿. Formally, x+ym=qrx + ym = qrx+ym=qr﻿ is satisfied for some positive integer yyy﻿. So, we can just keep adding mmm﻿ until we encounter a multiple of qqq﻿, which we expect to be qrqrqr﻿, then divide it by qqq﻿ to get rrr﻿. However, qrqrqr﻿ might not be the first multiple of qqq﻿ we encounter. When can this happen?

The answer is actually a bit complicated, but you may assume you can apply this method when qqq﻿ and mmm﻿ are coprime. In that case, the smallest multiple encountered is always going to be qrqrqr﻿.

We are not restricted to these methods anyway. Note that mmm﻿ is almost always taken 109+710^9 + 7109+7﻿ (a prime) especially when the calculations involve division.

There is a reason: letting Fermat’s little theorem be applicable!

Fermat’s little theorem
If ppp﻿ is a prime, then ap≡a(modp)a^p \equiv a \pmod pap≡a(modp)﻿ for any integer aaa﻿.
Moreover, we can write ap=kp+aa^p = kp + aap=kp+a﻿. If ppp﻿ does not divide aaa﻿, then kkk﻿ is definitely divisible by aaa﻿. So, ap−1=k′p+1a^{p-1} = k'p + 1ap−1=k′p+1﻿, and we get ap−1≡1(modp)a^{p-1} \equiv 1 \pmod pap−1≡1(modp)﻿.
Its generalized version is Euler’s theorem: aϕ(n)=1(modn)a^{\phi (n)} = 1 \pmod naϕ(n)=1(modn)﻿ if aaa﻿ and nnn﻿ are coprime.

Using this ap−1≡1(modp)a^{p-1} \equiv 1 \pmod pap−1≡1(modp)﻿ congruence, we can multiply any fraction mn\frac{m}{n}nm​﻿ by np−1n^{p-1}np−1﻿ and calculate m⋅np−2 mod pm \cdot n^{p-2} \bmod pm⋅np−2modp﻿ to find mn mod p\frac{m}{n} \bmod pnm​modp﻿. It’s just that nnn﻿ must not be divisible by ppp﻿, which is not likely to happen for p=109+7p = 10^9 + 7p=109+7﻿. This property is referred to as “inverse modulo”:

np−2n^{p-2}np−2﻿ is called the modular inverse of nnn﻿, because their product modulo mmm﻿ is 111﻿. This is very similar to multiplicative inverse (product itself results in 111﻿).

Hey, stop there! How do we even calculate np−2 mod pn^{p-2} \bmod pnp−2modp﻿ for p=109+7p = 10^9 + 7p=109+7﻿? We can’t multiply a billion nnn﻿’s in under a second!

Yes, that’s right… But don’t worry, there is a special method just to calculate that!

Calculating ab mod ma^b \bmod mabmodm﻿ FAST: Binary Exponentiation!

When bbb﻿ is small enough, like 101010﻿ million, we can multiply aaa﻿ 101010﻿ million times since each multiplication is a single operation.

But there are going to be lots of problems where bbb﻿ is large like 10910^9109﻿, or even huge like 101810^{18}1018﻿.

What to do then?

Forget about aba^bab﻿ itself for once. Starting with aaa﻿, we can multiply our value by itself in a single operation:

  1. a⋅a=a2a \cdot a = a^2a⋅a=a2﻿ (we now know a2a^2a2﻿)
  1. a2⋅a2=a4a^2 \cdot a^2 = a^4a2⋅a2=a4﻿ (now, we have a4a^4a4﻿)
  1. a4⋅a4=a8a^4 \cdot a^4 = a^8a4⋅a4=a8﻿
  1. a8⋅a8=a16a^8 \cdot a^8 = a^{16}a8⋅a8=a16﻿
  1. a16⋅a16=a32a^{16}\cdot a^{16} = a^{32}a16⋅a16=a32﻿

Can you see the pattern? Since we always double the exponent, they are in the following form:

We start from n=1n = 1n=1﻿, and at the ithi^{th}ith﻿ step, we calculate a2i\Large a^{2^{i}}a2i﻿. We can also denote 2i2^i2i﻿ with ttt﻿ and say at the (log⁡2t)th(\log_2 t)^{th}(log2​t)th﻿ step, we calculate ata^tat﻿.

So, it takes a logarithmic number of steps to calculate ata^{t}at﻿. In other words, its complexity is O(log⁡t)\mathcal{O}(\log t)O(logt)﻿.

Anyway, how do we use these powers to calculate aba^bab﻿?

For that, we use bbb﻿’s binary representation.

Binary representation

Binary representation represents any integer by summation of some powers of 222﻿. For example, 5=22+20=(101)2,29=24+23+22+20=(11101)25 = 2^2 + 2^0 = (101)_2, 29 = 2^4 + 2^3 + 2^2 + 2^0 = (11101)_25=22+20=(101)2​,29=24+23+22+20=(11101)2​﻿.

To obtain the binary representation of an integer, we can continuously divide it by 222﻿, and whenever the remainder is 111﻿, we include that power of 222﻿ (2i2^i2i﻿ after the (i+1)th(i+1)^{th}(i+1)th﻿ division) in the summation (representation). In fact, the 111﻿’s represent these remainders.

Let’s demonstrate it with a relatively large number 314314314﻿:

Thus, 314=28+25+24+23+21=(100111010)2314 = 2^8 + 2^5 + 2^4 + 2^3 + 2^1 = (100111010)_2314=28+25+24+23+21=(100111010)2​﻿.

For each power of 222﻿ that is present in bbb﻿’s binary representation, we can multiply the corresponding powers of aaa﻿ to calculate aba^bab﻿.

We do this in parallel to the calculation of aaa﻿’s powers. Let’s demonstrate with b=108=26+25+23+22b = 108 = 2^6 + 2^5 + 2^3 + 2^2b=108=26+25+23+22﻿:

  1. Initialize the answer to 111﻿. It will be aba^bab﻿ in the end.
  1. a⋅a=a2a \cdot a = a^2a⋅a=a2﻿ — 212^121﻿ is not taken.
  1. a2⋅a2=a22a^2 \cdot a^2 = a^{2^2}a2⋅a2=a22﻿— 222^222﻿ should be taken. Multiply the answer by a22a^{2^2}a22﻿. It is now a22a^{2^2}a22﻿.
  1. a22⋅a22=a23a^{2^2} \cdot a^{2^2} = a^{2^3}a22⋅a22=a23﻿ — 232^323﻿ should be taken. Multiply the answer by a23a^{2^3}a23﻿. It is now a22+23a^{2^2 + 2^3}a22+23﻿.
  1. a23⋅a23=a24a^{2^3} \cdot a^{2^3} = a^{2^4}a23⋅a23=a24﻿ — 242^424﻿ is not taken.
  1. a24⋅a24=a25a^{2^4} \cdot a^{2^4} = a^{2^5}a24⋅a24=a25﻿ — 252^525﻿ should be taken. Multiply the answer by a25a^{2^5}a25﻿. It is now a22+23+25a^{2^2 + 2^3 + 2^5}a22+23+25﻿.
  1. a25⋅a25=a26a^{2^5} \cdot a^{2^5} = a^{2^6}a25⋅a25=a26﻿ — 262^626﻿ should be taken. Multiply the answer by a26a^{2^6}a26﻿. It is now a22+23+25+26a^{2^2 + 2^3 + 2^5 + 2^6}a22+23+25+26﻿.
  1. a108a^{108}a108﻿ has been calculated.
Implementation

There might be some unfamiliar lines like result = 1ll * result * base % mod. 1ll is used for overflow prevention when both result and base are 32-bit integers. We will discuss them in the next section!

// https://ideone.com/AoHkEA
#include <bits/stdc++.h>
using namespace std;
using ll = long long;

const int mod = 1e9 + 7;

int binary_exponentiation(ll base, ll exponent, bool take_exponent_mod = false) {
    // The base can also be huge. Remember the properties of multiplication modulo m:
    //   We may take base % m beforehand since we just trying to compute base * base * base * ... * base (exponent times).
    base %= mod;

    // In fact, you can take exponent % (m - 1) — remember base^(m - 1) = 1 (mod m)
    if (take_exponent_mod)
        exponent %= mod - 1;

    int result = 1;
    // Find the powers of 2 in exponent's binary representation,
    // and simultaneously, multiply base with itself (base^2, base^4, base^8,...)
    // That is, they can be done simultaneously.
    while (exponent) {
        // 2's power is taken only if current value of exponent is odd!
        if (exponent & 1) { // bitwise AND, identical to exponent % 2
            result = 1ll * result * base % mod;
        }

        exponent /= 2; // or exponent >>= 1 (right shift, i.e., shift bits one to the right, drop the last bit)

        // Multiply base with itself to get the next value in [base^2, base^4, base^8,...]
        base = 1ll * base * base % mod;
    }

    return result;
}

int main() {
    ll base = 1e18, exponent = 1e18;
    cout << base << "^" << exponent << " modulo " << mod << ": " << binary_exponentiation(base, exponent) << "\n";
    cout << base << "^" << exponent % mod << " modulo " << mod << " (took exponent mod m): " << binary_exponentiation(base, exponent, true) << "\n";
}
# https://ideone.com/GQYCm3
def binary_exponentiation(base: int, exponent: int, mod: int, take_exponent_mod=False) -> int:
    result = 1
    while exponent:
        if exponent & 1:  # or exponent % 2
            result = result * base % mod
        exponent >>= 1  # or exponent /= 2
        base = base * base % mod
    return result


base = 10**18
exponent = 10**18
mod = 10**9 + 7

print(f"{base}^{exponent} modulo {mod}: {binary_exponentiation(base, exponent, mod)}")
print(
    f"{base}^{exponent % mod} modulo {mod} (took exponent mod m): {binary_exponentiation(base, exponent, mod, take_exponent_mod=True)}"
)

# Python has built-in binary exponentiation: pow(base, exponent, mod)!
print(f"{base}^{exponent} modulo {mod} (built-in): {pow(base, exponent, mod)}")
