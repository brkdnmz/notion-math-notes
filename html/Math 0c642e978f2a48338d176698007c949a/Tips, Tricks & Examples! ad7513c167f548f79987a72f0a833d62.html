Operator Precedence

In programming languages, some operators have precedence over others, that is, their operations are performed first.

For instance, multiplication precedes (is done before) addition and subtraction, and so does division: Firstly multiplication is performed, then addition.

To manipulate the precedence, we use parentheses (...): We can compute 363636﻿ by 4 * (5 + 3 + 1). Therefore, we don’t actually need to know which operator precedes which — we can just use parentheses. However, using parentheses everywhere is sometimes boring and causes dirtiness, at least in my opinion. So, knowing precedence is pretty useful.

You can see the precedences of operators in C++ here: https://en.cppreference.com/w/cpp/language/operator_precedence.

When several operators have the same precedence, their associativity is taken into account. Associativity just stands for in which direction an expression involving several operators is computed.

Notice that multiplication, division, and modulo (remainder) operations have the same precedence of 555﻿. Therefore, 3 * 4 % 5 * 8 % 11 is equivalent to (((3 * 4) % 5) * 8) % 11, which equals 555﻿.

Reference as Result: Assignment Operators

All operators can be considered functions. For instance, a * b is just like multiply(a, b).

Functions can take parameters’ references. A reference of a variable is the same variable but named differently. If you mutate a reference (change its value), you also change the variable it refers to. A reference to a variable typed t should be defined starting with t&. For int, it is int&.

You may see the exemplary codes below:

int a = 0;
int& b = a;
b += 5; // a = b = 5
b *= 3; // a = b = 15
b -= 7; // a = b = 8
void mul(int& a, int b){
	a *= b;
}

int a = 6;
mul(a, 3); // a is now 18

Some operators don’t return the result of the operation, but they return a reference to the left operand itself.

Most of them are assignment operators. They are often used for convenient coding, but their returned references can also be used for chaining operations.

Some examples are shown in the code below:

// https://ideone.com/5l0e2Z
#include <iostream>
using namespace std;
 
int main() {
	int a = 45;
	(((a -= 7) /= 4) *= 5) %= 11;
	cout << a << "\n";
}
  • After a -= 7, a becomes 383838﻿.
  • The left-hand side of the operation /= 4 is a -= 7 which returns a reference to a after subtracting 777﻿. Therefore, 38 / 4 is performed: a becomes 999﻿.
  • After a *= 5 (where a=9a = 9a=9﻿), a becomes 454545﻿.
  • After a %= 11, a becomes 111﻿.

It is more used in mod calculations, such as (a *= b) %= mod.

Overflow: How to Perform Mod Calculations Safely

In C++, we may just use long longs to prevent possible overflows that could be caused by using ints.

An int is bounded by 231−1=2147483647≈2⋅1092^{31} - 1 = 2147483647 \approx 2 \cdot 10^9231−1=2147483647≈2⋅109﻿, whereas long long can be up to 263−1=9223372036854775807≈9⋅10182^{63} - 1 = 9223372036854775807 \approx 9 \cdot 10^{18}263−1=9223372036854775807≈9⋅1018﻿. It’s because they are assigned 323232﻿ and 646464﻿ bits in the memory (the first bit is for sign).

Since they are bounded, they cannot have a larger value than their upper bound. If an operation results in a larger value, overflow occurs, just like how excess water overflows when a water glass is overfilled:

However, overflow doesn’t work the same way in programming: the glass doesn’t remain filled.

When integers overflow, they go back to the lower bound. For int, it is −231-2^{31}−231﻿ and for long long, it is −263-2^{63}−263﻿. (Notice that there are 2322^{32}232﻿ and 2642^{64}264﻿ integers within the boundaries, respectively.)

See the code below for an example:

// https://ideone.com/2fsoi7
#include <iostream>
using namespace std;
 
int main() {
	int a = 2147483647;
	cout <<"a: " << a << "\n";
	cout <<"a + 1: " << a + 1 << "\n";
	cout <<"a + 2: " << a + 2 << "\n";
	cout <<"2a: " << 2 * a << "\n";
	cout <<"a + a: " << a + a << "\n";
	cout <<"10a: " << 10 * a << "\n";
}

Output

a + 1: -2147483648

2a: -2

Therefore, performing modulo operations results within [0,109+7)[0, 10^9 + 7)[0,109+7)﻿. If we take such three integers a,b,ca, b, ca,b,c﻿ and  mod =109+7\bmod = 10^9 + 7mod=109+7﻿:

These might overflow:

// a b, c are both ints
a * b % mod;
(a + b + c) % mod;
a * b * c % mod;
a * a % mod;
a * b * a * b * a * b % mod;

Recall operator precedence: multiplication and modulo have the same precedence.

  • For a * b % mod, first, a * b is performed. Overflow might occur right there because they are both ints whereas the result might exceed the int bound as a,b<109+7a, b < 10^9 + 7a,b<109+7﻿.
  • For (a + b + c) % mod, first, a + b is performed. It’s not going to overflow since the result can be at most 2⋅( mod −1)=2⋅109+122 \cdot (\bmod - 1) = 2 \cdot 10^9 + 122⋅(mod−1)=2⋅109+12﻿. Then, + c is performed. Here, the actual result can be at most 3⋅( mod −1)3 \cdot (\bmod - 1)3⋅(mod−1)﻿ which surely exceeds the int bound.

The next examples are similar to these.

These are guaranteed to not overflow:

(a + b) % mod;
((a + b) % mod + c) % mod;
1ll * a * b % mod;
1LL * a * b % mod;
(long long)a * b % mod;
1ll * a * b % mod * c % mod;
1ll * a * b % mod * (1ll * b * c % mod);
1ll * a * b % mod * b % mod * b % mod * b % mod * b % mod;
  • (a + b) % mod cannot overflow, it is explained on the left.
  • The result of (a + b) % mod is also less than mod. Therefore, ((a + b) % mod + c) % mod is similar to the first example.
  • 1ll is a literal just like 1, 2, 3 standing for “111﻿ long long”, i.e., 646464﻿-bit 111﻿. Any expression containing long longs and ints can be treated as if each element were long long. Therefore, 1ll * a * b won’t overflow, and its mod can be taken safely.
  • 1LL is an alias for 1ll. They are identical.
  • Instead of adding a factor of 1ll, we can cast one of the variables to long long such as (long long) a.
  • 1ll * a * b % mod results in less than mod, and the result is also 64-bit. So, multiplying by c won’t hurt.
