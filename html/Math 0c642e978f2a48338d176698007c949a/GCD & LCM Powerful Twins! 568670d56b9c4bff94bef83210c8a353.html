GCD, a.k.a. Greatest Common Divisor

To define GCD, we need a group III﻿ of nnn﻿ integers i1,i2,…,ini_1, i_2, \dots, i_ni1​,i2​,…,in​﻿ such that at least one of them is non-zero. (nnn﻿ is often 222﻿.)

Ignore the “greatest” adjective, and focus on “common divisor”.

A common divisor of III﻿ is an integer ccc﻿ that divides each integer in III﻿.

So, for G={2,4,6}G = \{2, 4, 6\}G={2,4,6}﻿, there are 444﻿ common divisors: ±1,±2\pm 1, \pm 2±1,±2﻿.

But is there any reasonable method for finding a common divisor?

Of course! Remember that for an integer ddd﻿ to divide nnn﻿, each prime’s exponent in ddd﻿’s prime factorization should be less than or equal to that in nnn﻿’s prime factorization.

We can combine the inequalities above into one by using the min⁡\minmin﻿ function.

To find a proper ccc﻿, it is enough to choose an exponent fkf_kfk​﻿ for each prime pkp_kpk​﻿.

Now, put “greatest” back. How to find the greatest common divisor?

The answer is trivial now: choose the maximum possible fkf_kfk​﻿. Formally,

That is, for each prime, choose the minimum exponent in factorizations of integers in III﻿ — it’s what the exponent in GCD’s factorization is.
Note that you may ignore 000﻿ since it cannot ever contain the minimum exponent for any prime. However, if III﻿ only contains 000﻿’s, then GCD is considered undefined.

GCD can be calculated via iterations.

Inspect the min⁡\minmin﻿ function. Suppose we pass it nnn﻿ integers a1,a2,…,ana_1, a_2, \dots, a_na1​,a2​,…,an​﻿, i.e., we are trying to find min⁡{a1,a2,…,an}\min\{a_1, a_2, \dots, a_n\}min{a1​,a2​,…,an​}﻿. Notice that the answer (the minimum integer) does not change if we first split these nnn﻿ integers into ggg﻿ subgroups, then calculate min⁡\minmin﻿ of each subgroup, and finally calculate min⁡\minmin﻿ of these ggg﻿ min⁡\minmin﻿s. It’s because at least one of them will be equal to the overall minimum integer, and we’ll obtain it after the final min⁡\minmin﻿ calculation. For instance, let’s say n=6n = 6n=6﻿ and integers are 10,4,5,16,3,810, 4, 5, 16, 3, 810,4,5,16,3,8﻿ (min⁡=3\min = 3min=3﻿), and split them into 333﻿ groups: {10},{4,5,16},{3,8}\{10\}, \{4, 5, 16\}, \{3, 8\}{10},{4,5,16},{3,8}﻿. Then, the min⁡\minmin﻿s of these groups are 10,4,310, 4, 310,4,3﻿, respectively. Finally, min⁡{10,4,3}=3\min\{10, 4, 3\} = 3min{10,4,3}=3﻿. See? It’s the overall min⁡\minmin﻿.

The opposite of this also holds, that is, two subgroups can be merged into one. min⁡\minmin﻿ is not going to change.

How does this help?

GCD is pretty much all about calculating the min⁡\minmin﻿s of exponents in factorizations. So, we can calculate the GCD of III﻿ by iterating over the integers one by one. Let’s take an exemplary III﻿ and demonstrate this process.

GCD w/o Prime Factorization (Euclid’s Algorithm)

In my opinion, GCD is best understood via prime factorization due to selecting minimum exponents. However, some observations become much clearer when we ignore factorization.

Remember the “divisor expression”? It was n=dkn = dkn=dk﻿ where ddd﻿ is a divisor of nnn﻿. We can use it for GCD too!

From now on, let’s only deal with two integers — it will make our lives better.

Take two integers a,ba, ba,b﻿, and let GCD(a,b)=g\text{GCD}(a, b) = gGCD(a,b)=g﻿. Here are the observations we can make at first glance:

  • Since ggg﻿ is their common divisor, we can express them as a=g⋅xa = g \cdot xa=g⋅x﻿ and b=g⋅yb = g \cdot yb=g⋅y﻿.
  • Since ggg﻿ is their greatest common divisor, then GCD(x,y)=1\text{GCD}(x, y) = 1GCD(x,y)=1﻿. (This type of pairs are called coprime, because they don’t share a common divisor other than 111﻿.)
    Otherwise, if GCD(x,y)=g′>1\text{GCD}(x, y) = g' > 1GCD(x,y)=g′>1﻿, we could also write x=g′⋅x′x = g' \cdot x'x=g′⋅x′﻿ and y=g′⋅y′y = g' \cdot y'y=g′⋅y′﻿. Then, it would be the case that a=g⋅g′⋅x′a = g \cdot g' \cdot x'a=g⋅g′⋅x′﻿ and b=g⋅g′⋅y′b = g \cdot g' \cdot y'b=g⋅g′⋅y′﻿, then GCD(a,b)≥g⋅g′>gGCD(a, b) \ge g \cdot g' > gGCD(a,b)≥g⋅g′>g﻿. But we started by saying that GCD is equal to ggg﻿.

Here is the question:

Does GCD change at all when we subtract one number from another, e.g., subtract bbb﻿ from aaa﻿?

Let’s find out. a−b=g(x−y)a-b = g(x-y)a−b=g(x−y)﻿ and b=g⋅yb = g \cdot yb=g⋅y﻿. Both are divisible by ggg﻿, so their GCD is divisible by ggg﻿ (at least ggg﻿).

Let GCD(x−y,y)=g′\text{GCD}(x-y, y) = g'GCD(x−y,y)=g′﻿. Then, their sum xxx﻿ is also divisible by g′g'g′﻿. Thus, GCD(x,y)\text{GCD}(x, y)GCD(x,y)﻿ is at least g′g'g′﻿ because g′g'g′﻿ is a common divisor of xxx﻿ and yyy﻿.

But we know that GCD(x,y)=1\text{GCD}(x, y) = 1GCD(x,y)=1﻿. As a consequence, g′g'g′﻿ must be 111﻿. So, GCD(x−y,y)=1\text{GCD}(x-y, y) = 1GCD(x−y,y)=1﻿, and as they don’t share a common divisor, GCD(a−b,b)=g\text{GCD}(a-b, b) = gGCD(a−b,b)=g﻿.

The answer is, GCD does not change after subtraction. In fact, it also holds for addition.

How does this help?

Recall the modulo operation. It is used to find the remainder, and the remainder is basically the result of repeated subtractions a−b−b−b…a - b - b - b \dotsa−b−b−b…﻿

If GCD does not change after a single subtraction, then it does not also change after multiple subtractions. This lets us use the modulo operation to calculate GCD:

And, this is how GCD can be calculated so fast. We repeatedly apply the modulo operation on the bigger side (aaa﻿ if a>ba > ba>b﻿, otherwise bbb﻿) until one becomes zero. In that case, GCD is equal to the other.

Why did the third example take so many steps? It’s because the worst case occurs when aaa﻿ and bbb﻿ are consecutive Fibonacci numbers! (Figure out why)

Because of this, the time complexity of GCD calculation is O(log⁡min⁡(a,b))\mathcal{O}(\log \min(a, b))O(logmin(a,b))﻿.

This method is also known as the “Euclidean Algorithm”, which is referred to on Wikipedia.
Implementation
// https://ideone.com/MZLH0V
#include <bits/stdc++.h>
using namespace std;
using ll = long long;

ll gcd(ll a, ll b) {
    // always assign a = a % b for convenience
    if (a < b)
        swap(a, b); // built-in function

    while (b) { // while b is non-zero, continue the process
        a %= b;
        swap(a, b);
    }
    // at the end, a will be the GCD!
    return a;
}

ll gcd_recursive(ll a, ll b) {
    if (a < b)
        swap(a, b);
    if (!b)
        return a;
    return gcd_recursive(a % b, b);
}

// You don't need to swap yourself. This is an alternative version.
ll gcd_recursive_alt(ll a, ll b) {
    if (!a) // a will eventually become zero.
        return b;
    // First, change the parameters' location,
    // then apply modulo. This is totally the same.
    return gcd_recursive_alt(b % a, a);
}

int main() {
    ll a = 152348643862835682ll;
    ll b = 4236438563286243ll;
    cout << "GCD (iterative):\t" << gcd(a, b) << "\n";
    cout << "GCD (recursive):\t" << gcd_recursive(a, b) << "\n";
    cout << "GCD (recursive v2):\t" << gcd_recursive_alt(a, b) << "\n";

    // Guess what? There is a  built-in GCD function!
    cout << "GCD (built-in):\t\t" << __gcd(a, b) << "\n";
}

from math import gcd


def gcd_iterative(a: int, b: int) -> int:
    if a < b:
        a, b = b, a

    while b:
        a %= b
        a, b = b, a

    return a

def gcd_recursive(a: int, b: int) -> int:
    if a < b:
        a, b = b, a

    if not b:
        return a

    return gcd_recursive(a%b, b)

def gcd_recursive_alt(a: int, b: int) -> int:
    if not a:
        return b

    return gcd_recursive_alt(b%a, a)

a = 152348643862835682
b = 4236438563286243

print(f"GCD (iterative):\t{gcd_iterative(a, b)}")
print(f"GCD (recursive):\t{gcd_recursive(a, b)}")
print(f"GCD (recursive v2):\t{gcd_recursive_alt(a, b)}")
print(f"GCD (built-in):\t\t{gcd(a, b)}")

LCM, a.ka. Least Common Multiple: GCD’s Elder Brother

As its name describes, LCM is the least of positive common multiples. Remember what we did with prime factorizations to find GCD: we chose the minimum exponent for each prime.

For LCM, it’s very similar: we choose the maximum exponent for each prime.

Because it gives us the smallest common multiple possible among infinitely many common multiples.

There isn’t really much else to explain. 😃 It can be defined formally similar to how we defined GCD:

If a group of integers contains 000﻿, unlike GCD, LCM cannot be computed since 000﻿ does not have a multiple other than itself. This isn’t a logical scenario anyway.

LCM has the same “splittable” property, similar to GCD. We can calculate LCM in any way we want.

A special case: GCD and LCM of 222﻿ integers

Take any two integers aaa﻿ and bbb﻿. Notice that for GCD, we take the smaller exponent and for LCM, we take the bigger, thus, the other exponent from prime factorizations.

Once we wrote that down, we can observe that the sum of exponents in GCD and LCM is equal to the sum of exponents in aaa﻿ and bbb﻿.
